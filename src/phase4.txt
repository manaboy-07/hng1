# 🔐 AUTHENTICATION (MOST IMPORTANT PART)

## 🧠 Requirement:

> Use HTTP-only cookies
> Tokens must NOT be accessible via JS
> Include CSRF protection

---

## 🔥 What this means in practice

### ❌ OLD (CLI / basic frontend)

```js
localStorage.setItem("access_token")
```

👉 insecure

---

### ✅ NEW (Web Portal)

```txt
Browser ← HttpOnly Cookie ← Backend
```

👉 You DO NOT store tokens in JS

---

## 🔧 Backend change (important)

Instead of:

```ts
return { access_token, refresh_token }
```

👉 You now do:

```ts
res.cookie('access_token', access_token, {
  httpOnly: true,
  secure: true,
  sameSite: 'strict',
});

res.cookie('refresh_token', refresh_token, {
  httpOnly: true,
  secure: true,
  sameSite: 'strict',
});
```

---

## 🧠 Flow now becomes:

```txt
Login → cookies set
Browser → automatically sends cookies
Backend → reads cookies
```

---

# 🛡️ CSRF Protection (don’t skip this)

Because cookies are auto-sent → vulnerable to CSRF.

---

## ✅ Simple implementation

### Backend:

```ts
import * as csurf from 'csurf';

app.use(csurf());
```

---

### Frontend:

Send CSRF token:

```js
fetch('/api/profiles', {
  method: 'POST',
  headers: {
    'X-CSRF-Token': token,
  },
});
```

```txt

(no token manually added)

```

---

## 🧠 Logout

```ts
POST /auth/logout
```

Backend clears cookies:

```ts
res.clearCookie('access_token');
res.clearCookie('refresh_token');
```

---

# 🔁 Refresh Token (Web version)

👉 You don’t manually handle it in frontend

Backend does:

* read refresh cookie
* issue new access cookie

# 🧠 VERY IMPORTANT DIFFERENCE

| CLI             | Web                  |
| --------------- | -------------------- |
| Manual tokens   | Cookies              |
| JS handles auth | Backend handles auth |
| Visible tokens  | Hidden tokens        |

# 🔥 Protect routes (important)

Use middleware:

```ts
// middleware.ts
export function middleware(req) {
  const token = req.cookies.get('access_token');

  if (!token) {
    return NextResponse.redirect('/login');
  }
}
```
